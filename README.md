# PROMOCODE-SERVICE

## Задача

Реализовать backend-систему промокодов для подписочного digital-продукта.

Промокод может давать:

- скидку в процентах
- фиксированную скидку
- бонусные дни / продление подписки
- пополнение внутреннего баланса / бонусов

Что важно предусмотреть:

1. У промокода должны быть базовые поля:

- сам код
- тип бонуса
- значение бонуса
- дата начала и окончания действия
- активен / неактивен
- лимит общего числа использований
- лимит использований на одного пользователя
- одноразовый или многоразовый
- персональный или публичный
- привязка к конкретному пользователю, если промокод персональный
- внутренний комментарий / описание

2. Нужна логика ограничений:

- только для новых пользователей
- только для определенных тарифов
- запрет повторного использования
- запрет после окончания срока действия
- запрет после достижения лимита
- запрет, если пользователь не подходит под условия

3. Нужна проверка валидности промокода:

- существует ли он
- активен ли
- не истек ли срок
- не превышен ли лимит
- подходит ли пользователю
- применим ли к выбранному тарифу / заказу

4. После применения система должна:

- зафиксировать факт использования
- сохранить, кто использовал, когда и к чему применил
- вернуть корректный результат применения
- не допускать двойного применения

Что нужно сделать по backend:

- создание промокода
- список промокодов
- просмотр одного промокода
- проверка промокода перед применением
- применение промокода
- история использования промокода

По стеку предпочтительно:
Python + Django / DRF

Можно без фронтенда и без реальной платежки. Достаточно backend-логики, базы и API. Проверку можно показать через Postman / Swagger / Django admin.

Что нужно прислать по результату:

- код
- README с описанием запуска
- список endpoint’ов
- краткое объяснение архитектуры
- примеры запросов

Что будет плюсом:

- Django admin
- unit-тесты
- защита от повторного применения / race condition
- аккуратная структура моделей и сервисного слоя


## Допущения
- код уникален и соответствует регулярному выражению `^[A-Z0-9]{6,12}$`
- для значения бонуса могут быть использованы не целые числа, поэтому для хранения исопльзуется Decimal
- промокод может не иметь лимита активаций
- поле "одноразовый или многоразовый" избыточно, достаточно использовать "лимит общего числа использований"
- запрет повторного использования реализован засчёт лимита использований для одного пользователя
- пользователи хранятся во внешний системе и связаны через UUID
- дата начала и окончания действия не включены в период активности (строгие проверки > и <) 
- REST API не требует авторизации

## Запуск проекта

```bash
uv sync
source .venv/bin/activate
# или
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

python manage.py migrate
python manage.py createsuperuser
python manage.py runserver
```

## Архитектура

### Структура приложения

- Сущности [service/models](service/models)
- Бизнес логика [service/services](service/services)
- Внешнее API [service/views.py](service/views.py)

### Сущности

- Основные (бизнес логика)
- - Тариф
- - Промокод
- - Активация промокда
- Служебные
- - UserPromocodeUsage - количество активаций промокода пользователем
- - UserFirstUsage - первая активация промокода пользователя

### Обработка конкурентности реализована за счет

- блокировок в момент активации промокода (`SELECT FOR UPDATE`)
- unique индексов на служебных таблицах
- инкрементальному увеличению счетчиков
